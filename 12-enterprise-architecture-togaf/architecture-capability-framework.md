# Architecture Capability Framework

## Introduction

The Architecture Capability Framework addresses the organization, processes, skills, roles, and responsibilities required to establish and operate architecture practice.

## Key Concepts

| Concept | Definition |
|---------|------------|
| **Architecture Capability** | Ability of organization to perform architecture work effectively |
| **Architecture Maturity** | Measure of how well architecture practice is established |
| **Architecture Skills Framework** | Roles, skills, and experience required for architecture function |
| **Architecture Board** | Governance body for architecture decisions |

---

## Q&A — Architecture Capability

### Basic

**Q: What are the key components of an Architecture Capability?**

**A:** Key Components:

**1. Architecture Board:**
- Governance and decision-making body
- Cross-functional representation
- Authority to approve architectures

**2. Architecture Compliance:**
- Process for ensuring conformance
- Review mechanisms
- Dispensation process

**3. Architecture Repository:**
- Storage for architecture assets
- Standards and reference models
- Governance log

**4. Architecture Skills:**
- Trained architects
- Defined roles and responsibilities
- Career development paths

**5. Architecture Method:**
- Tailored ADM
- Tools and templates
- Best practices

**6. Architecture Principles:**
- Guiding rules for architecture decisions
- Business and technical principles
- Conflict resolution mechanisms

---

**Q: What are common architecture roles and their responsibilities?**

**A:**

**Enterprise Architect:**
- Define enterprise architecture strategy
- Lead Architecture Board
- Establish architecture standards
- Align architecture with business strategy

**Domain Architects:**
- Business Architect: Business processes, capabilities
- Data Architect: Data models, data governance
- Application Architect: Application portfolio, integrations
- Technology Architect: Infrastructure, platforms, networks

**Solution Architect:**
- Design solutions for specific projects
- Ensure alignment with enterprise architecture
- Bridge between project and EA team

**Architecture Analyst:**
- Support architects with research, modeling
- Maintain architecture repository
- Prepare documentation

---

**Q: What is Architecture Maturity and why does it matter?**

**A:** Architecture Maturity measures how well architecture practice is embedded in the organization.

**Maturity Levels (example model):**

**Level 1 - Initial:**
- Ad-hoc architecture
- No formal process
- Architecture done by developers

**Level 2 - Under Development:**
- Architecture team forming
- Some standards emerging
- Inconsistent adoption

**Level 3 - Defined:**
- Architecture framework established
- Standard processes
- Architecture governance in place

**Level 4 - Managed:**
- Metrics-driven architecture
- Proactive planning
- Strong stakeholder engagement

**Level 5 - Optimizing:**
- Continuous improvement
- Industry-leading practice
- Architecture innovation

**Why It Matters:**
- Higher maturity → Better business outcomes
- Enables transformation capability
- Reduces technical debt
- Improves ROI on IT investments

### Medium

**Q: How do you build an architecture capability from scratch in an organization that has never had formal EA?**

**A:** Building Architecture Capability - Phased Approach:

**Phase 1: Foundation (Months 1-3)**

**Quick Wins:**
- Appoint Chief/Lead Architect
- Document current state (major systems, integrations)
- Identify top 3 pain points (duplicate systems, integration chaos, tech debt)
- Create initial architecture principles (10-15 principles)

**Governance:**
- Form interim Architecture Review Board
- Define lightweight review process for new projects

**Communication:**
- Secure executive sponsorship
- Announce architecture initiative to organization

**Phase 2: Establishment (Months 4-9)**

**Team Building:**
- Hire domain architects (Business, Data, Application, Tech)
- Train existing staff in TOGAF

**Framework:**
- Tailor TOGAF ADM to organization
- Create templates and tools
- Establish architecture repository

**Standards:**
- Define technology standards (approved languages, platforms)
- Integration standards (API guidelines)
- Data standards (naming conventions, canonical models)

**Governance:**
- Formalize Architecture Board
- Implement compliance process for all new projects
- Track metrics (compliance rate, dispensations)

**Phase 3: Operationalization (Months 10-18)**

**Portfolio Management:**
- Application portfolio assessment
- Rationalization roadmap
- Business capability map

**Target Architecture:**
- Define enterprise target architecture
- Transition roadmaps
- Investment priorities

**Embedding:**
- Architecture checkpoints in project lifecycle
- Architecture training for project teams
- Success stories and showcases

**Phase 4: Optimization (Months 19+)**

**Maturity:**
- Measure architecture maturity
- Benchmark against industry
- Continuous improvement

**Innovation:**
- Architecture community of practice
- Research emerging technologies
- Pilot programs

**Metrics Throughout:**
- Stakeholder satisfaction
- Project compliance rate
- Technical debt reduction
- Time/cost savings from reuse

**Q: Design a skills framework for an enterprise architecture team.**

**A:** Enterprise Architecture Skills Framework:

**1. Role Definitions:**

**Enterprise Architect (Senior Level):**

**Technical Skills:**
- Deep expertise in multiple domains (Business, Data, Application, Technology)
- TOGAF certification
- Cloud architectures (AWS/Azure/GCP)
- Integration patterns and technologies
- Security and compliance

**Business Skills:**
- Business strategy and planning
- Financial analysis (TCO, ROI, business case)
- Industry knowledge
- Stakeholder management

**Soft Skills:**
- Leadership and influence
- Communication (executive presentations, technical documentation)
- Negotiation and conflict resolution
- Strategic thinking

**Experience:** 15+ years IT, 5+ years architecture

---

**Domain Architect (Mid-Senior Level):**

**Specialized Skills:**
- Expert in specific domain (Data, Application, etc.)
- Modeling techniques (UML, ArchiMate, BPMN)
- Domain-specific tools
- Standards and best practices for domain

**Cross-Domain Skills:**
- Understanding of other domains
- Integration points
- Impact analysis

**Experience:** 10+ years IT, 3+ years domain expertise

---

**Solution Architect (Mid Level):**

**Technical Skills:**
- Design patterns and principles
- Multiple technology stacks
- DevOps and CI/CD
- Performance and scalability

**Project Skills:**
- Requirements analysis
- Solution design documentation
- Vendor evaluation
- Technical leadership

**Experience:** 8+ years development, 2+ years architecture

---

**2. Competency Matrix:**

| Competency | EA | Domain Architect | Solution Architect |
|------------|-------|-----------------|-------------------|
| Business Strategy | Expert | Advanced | Intermediate |
| Technical Breadth | Expert | Advanced | Advanced |
| Technical Depth | Advanced | Expert | Expert |
| TOGAF/EA Frameworks | Expert | Advanced | Intermediate |
| Stakeholder Management | Expert | Advanced | Intermediate |
| Tool Proficiency | Advanced | Expert | Expert |

**Proficiency Levels:**
- Expert: Can teach others, innovate
- Advanced: Can work independently, mentor
- Intermediate: Can work with guidance
- Beginner: Learning

---

**3. Career Progression:**

**Path 1: Specialist Track**
- Developer → Senior Developer → Tech Lead → Solution Architect → Domain Architect → Principal Architect

**Path 2: Generalist Track**
- Solution Architect → Senior Solution Architect → Enterprise Architect → Chief Architect

**Path 3: Management Track**
- Domain Architect → Architecture Manager → Director of Architecture → VP Architecture

### Complex

**Q: An organization's architecture team is seen as a bottleneck and "ivory tower." How do you transform the perception and improve effectiveness?**

**A:** Transformation Strategy:

**1. Diagnose Root Causes:**

**Common Issues:**
- Lengthy review cycles
- Rejected proposals without alternatives
- Architecture not involved early enough
- Decisions perceived as arbitrary
- Lack of transparency

**Data Gathering:**
- Survey project teams
- Review turnaround times
- Analyze dispensation requests
- Stakeholder interviews

**2. Quick Wins (First 30 Days):**

**Accelerate Reviews:**
- Set SLA: 5 business days for standard reviews
- Create fast track for low-risk projects (self-service checklist)
- Dedicated review slots (daily architecture office hours)

**Communication:**
- Publish architecture principles with rationale
- Create FAQ and decision log
- Regular architecture updates (newsletter, blog)

**Engagement:**
- Attend project kickoffs (proactive, not reactive)
- Embed architects in major programs
- Architecture concierge service

**3. Shift Left (Architecture Earlier):**

**Phases of Engagement:**
- **Ideation:** Architect participates in brainstorming
- **Planning:** Architecture input on feasibility and approach
- **Design:** Collaborative design sessions
- **Implementation:** Lightweight checkpoints
- **Post-Implementation:** Capture lessons learned

**Benefit:** Avoid late-stage rejections; build solutions together

**4. Enable, Don't Block:**

**From "No" to "Yes, If":**
- Don't just reject; provide alternatives
- Create reference implementations
- Self-service blueprints and templates
- Inner source repository of reusable components

**Example:**
- Old: "You can't use MongoDB" (rejection)
- New: "We don't have MongoDB in standards. Here's how to use PostgreSQL JSON for your use case. If you still need MongoDB, here's the dispensation process and we'll help you." (guidance + path forward)

**5. Transparency:**

**Open Architecture:**
- Architecture decision records (ADRs) publicly accessible
- Rationale for standards documented
- Governance metrics published (compliance rates, turnaround times)
- Architecture Board meeting notes shared

**Feedback Loops:**
- Post-review surveys
- Quarterly retrospectives with project teams
- Architecture effectiveness metrics

**6. Build Partnerships:**

**With Development Teams:**
- Architecture champions in each team
- Invite developers to architecture discussions
- Co-create standards (not dictate)

**With Product/Business:**
- Business-focused architecture (value, not just technology)
- Join product planning meetings
- Speak business language (outcomes, not technical jargon)

**7. Measure Change:**

**Before/After Metrics:**
- Stakeholder satisfaction: 40% → 75%
- Review turnaround time: 3 weeks → 1 week
- Compliance rate: 60% → 90%
- Dispensation requests: 20% → 5%
- Projects with architecture involvement from start: 30% → 80%

**8. Cultural Shift:**

**From:** "Architecture team as gatekeepers"
**To:** "Architecture team as enablers"

**Behaviors:**
- Ask "How can we make this work?" not "Why won't this work?"
- Be pragmatic: perfect is the enemy of good
- Celebrate project successes (not just compliance)
- Admit mistakes and adapt

**9. Skills Development:**

**For Architects:**
- Empathy training (understand project pressures)
- Agile/DevOps mindset
- Negotiation and influence skills
- "Customer service" mindset

**10. Governance Evolution:**

**Lightweight Governance:**
- Automated compliance checks (shift-right to CI/CD)
- Risk-based reviews (high-risk = thorough, low-risk = fast)
- Trust but verify (spot checks vs. 100% reviews)

**Example:**
- Projects using only approved building blocks: automated approval
- Projects with new technology: architecture review
- Critical/high-risk projects: Architecture Board review

**Result:** Architecture team transforms from bottleneck to strategic partner, accelerating delivery while maintaining quality.
